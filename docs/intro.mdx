## Introduction to OLake
 

**OLake** is a blazing-fast open-source platform for both data replication and Iceberg table maintenance, moving data from diverse sources into Apache Iceberg and Parquet while keeping those tables optimized for real-time lakehouse analytics, without the pain of ETL scripts or vendor lock-in.

**GitHub Repository:** [https://github.com/datazip-inc/olake](https://github.com/datazip-inc/olake)  


## What is OLake?  

OLake is an open-source ELT platform, fully written in Golang for memory efficiency and high performance, that combines high-speed data replication with **[Iceberg table maintenance](/docs/iceberg-maintenance/overview)**. It replicates data from sources like PostgreSQL, MySQL, MongoDB, Oracle and Kafka directly into open lakehouse formats such as [Apache Iceberg](/iceberg/why-iceberg) and Parquet.
Using Incremental Sync and Change Data Capture (CDC), OLake keeps data continuously in sync while minimizing infrastructure overhead.

This allows organizations to:  
- Replicate data at scale  
- Power near real-time analytics  
- Transform data lakes into fully functional lakehouses without the overhead of complex ETL tools  
- Optimize storage costs and query performance.

---
## Why OLake?  

- **Fastest Path to a Lakehouse** → Achieve high throughput with [**parallelized chunking**](/docs/features/#1-parallelised-chunking) and **resumable** historical snapshots and blazing-fast incremental updates, even on massive datasets with **exactly-once** delivery.  

- **Efficient Data Capture** → Capture data efficiently with a full snapshot of your tables or collections, then keep them in sync through near real-time **CDC** using native database logs (**pgoutput, [binlogs](/blog/binlogs), [oplogs](/docs/understanding/terminologies/general/#26-oplog-mongodb)**).  

- **Schema-Aware Replication** → Automatically detect schema changes to keep your pipelines consistent and reliable.  

- **Open by Design** → Store data in **open formats** like Parquet and Iceberg, enabling **engine-agnostic analytics** and eliminating vendor lock-in.  

- **Iceberg table maintenance** → Keep lakehouse tables healthy after ingestion with scheduled optimization, better query performance, and lower storage costs.

## OLake Features Overview  

### Source-Level Features  

#### Supported Connectors  

- **PostgreSQL** → Supports **Full Refresh**, **Incremental Sync**, and **Pgoutput-based Full Refresh + CDC** and **Strict CDC** (RDS, Aurora, Supabase, etc.)  
- **MySQL** → Supports **Full Refresh**, **Incremental Sync**, and **Binlog-based Full Refresh + CDC** and **Strict CDC** (MySQL RDS, Aurora, older community versions)  
- **MongoDB** → Supports **Full Refresh**, **Incremental Sync**, and **Oplog-based Full Refresh + CDC** and **Strict CDC** (sharded or replica-set clusters)  
- **Oracle** → Supports **Full Refresh** and **Incremental Sync**  
- **Kafka** → Supports **Consumer Group Based Streaming (Append Only)** 

#### Optimized Chunking Strategies  

- **PostgreSQL** → CTID ranges, batch-size splits, next-query paging  
- **MySQL** → Range splits with LIMIT/OFFSET  
- **MongoDB** → Split-Vector, Bucket-Auto, Timestamp  
- **Oracle** → DBMS Parallel Execute  
- **Kafka** → Partition level reading via consumer group, each consumer processes the partitions assigned to it

### Iceberg Table Maintenance  

OLake is not limited to replication: it includes **Iceberg table maintenance** so you can configure and run optimization workflows in the same product you use for CDC and ingestion.

- **[Catalogs](/docs/iceberg-maintenance/catalogs)** — Register and validate Iceberg catalogs (including OLake-imported catalogs from ingestion) so tables are available for maintenance workflows.  
- **[optimization](/docs/iceberg-maintenance/optimization/overview)** — Configure scheduled **Lite**, **Medium**, and **Full** optimizations to reduce small files, merge delete files, and keep query performance predictable as data grows. Details: [optimization configuration](/docs/iceberg-maintenance/optimization/configuration).  
- **[Logs & Runs](/docs/iceberg-maintenance/runs-and-logs)** — Inspect each maintenance run and open task-level logs for troubleshooting.  
- **[Metrics](/docs/iceberg-maintenance/metrics)** — Review table and run metrics to see file counts, sizes, and the effect of each optimization.  

Start with the [Iceberg Table Maintenance overview](/docs/iceberg-maintenance/overview) or walk through [Configure your first optimization](/docs/getting-started/configure-first-optimization).
