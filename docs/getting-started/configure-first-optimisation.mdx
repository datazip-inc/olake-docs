# Configure Your First Optimisation

## Prerequisites

Follow the [Quickstart Setup Guide](/docs/getting-started/setup-olake) to ensure the [OLake UI](/docs/install/olake-ui/) is running at [localhost:8000](http://localhost:8000).

- You have at least one ingestion job writing data to an Iceberg destination **or**
- You are ready to add a catalog manually in the Optimisation section.

This guide walks through configuring your **first optimisation job** for a table.

## Step 1: Add a Catalog

Catalogs tell OLake where your Iceberg tables live.

:::info When you can skip this step
If you already have at least one ingestion job configured, and you can see the appropriate catalog under **Optimisation → Catalogs**, you can skip this step and move on to the Tables page.

You can learn more about OLake Managed Catalogs [here](/docs/iceberg-maintenance/catalogs/#olake-managed-catalogs).
:::

1. In the OLake UI sidebar, open the **Optimisation** dropdown and go to the **Catalogs** page.
2. Select **New Catalog**.

![Add catalog view](pathname:///img/docs/getting-started/configure-your-first-optimisation/add-catalog.webp)

3. Enter the catalog details.
4. Click on **Connect** to save and validate the catalog.

![Catalog connected view](pathname:///img/docs/getting-started/configure-your-first-optimisation/connect-catalog.webp)

For more details on Catalogs, see the [Catalogs documentation](/docs/iceberg-maintenance/catalogs/).


## Step 2: Select catalog and database

1. In the OLake UI sidebar, open the **Optimisation** dropdown and select **Tables**.
2. Use the **Select Catalog** dropdown and select the catalog you just configured.

![Catalog connected view](pathname:///img/docs/getting-started/configure-your-first-optimisation/select-catalog.webp)

3. After selecting the catalog, use the **Select Database** dropdown to choose a database from that catalog.

![Catalog connected view](pathname:///img/docs/getting-started/configure-your-first-optimisation/select-database.webp)

Only after you select both a catalog and a database will the list of tables for that database appear on the page.

## Step 3: Configure optimisation for your table

1. In the tables list, find the table you want to optimise.
2. Click on the **Configure** button.

![Configuration Button](pathname:///img/docs/getting-started/configure-your-first-optimisation/configure-button.webp)

This opens a configuration modal (cron modal) where you can set schedules for **Minor**, **Major**, and **Full** optimisation.

![Configuration Cron](pathname:///img/docs/getting-started/configure-your-first-optimisation/configure-cron.webp)

### Frequency presets

For each optimisation type, there is a **Frequency** dropdown with common schedules, such as:

- Every 30 min  
- Every hour  
- Every 8 hours  
- Every 12 hours  
- Every 24 hours  

You can configure these frequencies independently for Minor, Major, and Full optimisation.

### Custom cron expression

If you choose **Custom** in the Frequency dropdown, a **Cron Expression** field appears.

You can enter a standard cron expression here. For example:

- `0 0 * * *` – run the optimisation once every day at midnight.

## Step 4: (Advanced) Target file size

Under the **Advanced Config** dropdown in the modal, you can configure **Target file size**.

Target file size controls the **desired size of output files**:

- Larger target sizes usually mean **fewer, bigger files**.
- Smaller target sizes usually mean **more, smaller files**.

If you are unsure, you can start with the default value (**128mb**) and tune it later based on how your query engine performs.

## Step 5: Save the optimisation configuration

After configuring the cron:

1. In the modal, click on **Save**.
2. The configuration for that table is saved and a success state is shown, which auto-closes the modal in a few seconds.

![Save Configuration](pathname:///img/docs/getting-started/configure-your-first-optimisation/save-configuration.webp)

## Step 6: Enable the optimisation

Saving the configuration does not start optimisation automatically. You must enable it:

1. On the **Tables** page, locate the **Status** column next to the **Configure** button for your table.
2. Use the **toggle** in the **Status** column to enable the optimisation configuration.

![Enable Configuration](pathname:///img/docs/getting-started/configure-your-first-optimisation/enable-optimisation.webp)

Once enabled, OLake will start running compaction jobs for that table according to the schedule you configured.

## Understanding the Tables page columns

With a catalog and database selected, the **Tables** page shows one row per table. The sections below explain **Health Score** (overall table health) and **Last Run status** (per-type status for Lite, Medium, and Full optimisation).

### Health Score

**Health Score** is a single number that summarises how “healthy” the table looks from OLake’s perspective. It is **calculated** as:

**Health Score** = Small Files Score + Eq Delete Score + Pos Delete Score

- **Small Files Score** — reflects how fragmented or small your data files are.  
- **Eq Delete Score** — reflects equality delete files.  
- **Pos Delete Score** — reflects positional delete files.  

Higher scores generally mean the table is in better shape for reads; lower scores suggest running or tuning optimisation more often.

### Last Run status

**Last Run status** shows **all three** optimisation kinds at once—**Lite**, **Medium**, and **Full**—not only the single type that ran most recently. You get one indicator per type (**L**, **M**, **F**). Each indicator reflects the **latest run of that type** for the table: whether it **succeeded**, **failed**, was **cancelled**, or is still **running**.

- **Letters** — Typically you see three badges together:
  - **L** — Lite optimisation
  - **M** — Medium optimisation
  - **F** — Full optimisation

- **Colours (quick read, per badge)**  
  - **Green** — that type’s last run **succeeded**  
  - **Red** — that type’s last run **failed** or was **cancelled**  
  - **Yellow** — a run of **that type** is **in progress** right now  

- **Not Optimised** — Shown **only** when **no** optimisation has run yet for that table—**neither** Lite, **nor** Medium, **nor** Full.

:::info Hover
When you **hover** over a table's Last Run Status, a small **card** opens for **that** optimisation type with:

- **Name** — Lite, Medium, or Full  
- **Last run** — relative time, such as “2 hours ago”  
- **Outcome** — plain text such as **Success**, **Failed**, **Cancelled**, or **Running**
:::
