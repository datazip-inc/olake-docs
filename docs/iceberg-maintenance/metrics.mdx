# Overview

The Metrics view helps you understand both the **current health of an Iceberg table** and the **impact of optimisation runs**.

There are two types of metrics:

- **Table Metrics** – always available for a table, showing its current state.
- **Run Metrics** – available after an optimisation run finishes successfully.

## Table Metrics

You can see table-level metrics from the **Tables** page by clicking the **View Metrics** button for a table.

![Table metrics button](pathname:///img/docs/iceberg-maintenance/metrics/view-table-metrics-button.webp)

These metrics describe the current layout and size of the Iceberg table:

![Table metrics view](pathname:///img/docs/iceberg-maintenance/metrics/table-metrics.webp)

| Metric             | Description                                                                 |
|--------------------|-----------------------------------------------------------------------------|
| File Count         | Total number of data and delete files that currently make up the table.    |
| Total Size         | Combined on-disk size of all files that belong to the table.               |
| Average File Size  | Average size of files; useful to see if the table has many small files.    |
| Last Commit Time   | Time of the last successful Iceberg commit for the table.                  |
| Data Files Count   | Number of data files that store actual table rows.                         |
| Delete Files Count | Number of delete files that track removed or filtered-out rows.            |
| Health Score       | Combined score based on Small Files Score, Eq Delete Score, and Pos Delete Score. |

## Run Metrics

Run Metrics are shown after an optimisation run finishes successfully and are accessed from the **Runs** page via the **View Metrics** action for that run.

![Run metrics view](pathname:///img/docs/iceberg-maintenance/metrics/view-run-metrics-button.webp)

These metrics help you see how much a specific run improved the table:

![Run metrics view](pathname:///img/docs/iceberg-maintenance/metrics/run-metrics.webp)


| Metric          | Example           | Description                                                      |
|-----------------|-------------------|------------------------------------------------------------------|
| Runtime         | `30m`             | How long the optimisation run took from start to finish.        |
| Files Optimised | `192 → 187`       | Change in the number of files after the run.                    |
| Size Optimised  | `486 MB → 420 MB` | Change in the total table size on disk after the run.           |


## Interpreting Metrics

- A high **Delete Files Count** or very low **Average File Size** can indicate fragmentation and metadata overhead.
- After a successful optimisation run, **Files Optimised** and **Size Optimised** show how much the table improved.
- A low **Health Score** (driven by Small Files, Eq Deletes, and Pos Deletes) is a strong signal that the table can benefit from optimisation.
