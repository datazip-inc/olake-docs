# Optimization Configuration

Optimization in Iceberg is controlled through specific table properties and job parameters. These settings allow you to tune how data is compacted, how long history is retained, and how metadata is organized.

## Table Properties for Compaction

These properties influence how Iceberg engines handle data file creation and subsequent rewrite operations. Adjusting these helps balance write latency versus read efficiency.

| Property | Default | Description |
| :--- | :--- | :--- |
| `write.target-file-size-bytes` | `536870912` (512 MB) | The target size for data files after compaction. Larger files improve read performance for analytical queries. |
| `write.metadata.previous-versions-max` | `100` | The maximum number of previous manifest lists to keep in the metadata history. |
| `write.distribution-mode` | `none` | Defines how data is distributed across writers (e.g., `hash`, `range`). Correct distribution reduces the number of small files produced during ingestion. |

## Snapshot Expiration Settings

To prevent storage bloat, you must configure how aggressively old snapshots are removed. This is critical for managing cloud storage costs (S3/GCS/Azure).

| Property | Default | Description |
| :--- | :--- | :--- |
| `history.expire.max-snapshot-age-ms` | `432000000` (5 days) | Snapshots older than this threshold are eligible for expiration. |
| `history.expire.min-snapshots-to-keep` | `1` | The minimum number of snapshots to retain, regardless of age, to ensure point-in-time recovery. |

## Manifest & Metadata Optimization

The efficiency of query planning depends on the organization of manifest files. Merging manifests speeds up the metadata scan phase.

| Property | Default | Description |
| :--- | :--- | :--- |
| `commit.manifest.target-size-bytes` | `8388608` (8 MB) | Target size for manifest files. Optimizing this reduces the overhead of scanning metadata during query planning. |
| `commit.manifest-merge.enabled` | `true` | Whether to automatically merge small manifest files during commits. |

## Example: Updating Table Properties

You can apply these settings using SQL command directly on your Iceberg tables:

```sql
ALTER TABLE prod.db.my_table 
SET TBLPROPERTIES (
  'write.target-file-size-bytes' = '134217728',
  'history.expire.max-snapshot-age-ms' = '86400000'
);
```

> [!TIP]
> While these properties configure the *behavior* of maintenance, you still need to schedule the actual maintenance jobs (like `expireSnapshots` or `rewriteDataFiles`) using a scheduler or maintenance service.
